# User Stories: spec-lens-timeline

> このファイルは、`requirements.md` と `design.md` を読む前に、利用者目線の流れをつかむための補助資料である。
> ユーザーストーリーは「誰が」「何をしたくて」「なぜ必要か」を短く表す。実装方法やファイル構成はここでは主目的にしない。

## Product Story

SpecLens Timelineは、発表者がReact/Viteアプリをcc-sddで開発する過程を、あとから週単位のストーリーとして見返せるローカルアプリである。

発表者は、仕様変更、設計、タスク進行、ViteのHMR、エラー、検証、意思決定、スクリーンショットを1つのタイムラインで確認し、Viteカンファレンスで「どう作ったか」と「作ったアプリが何を解決するか」を同じデモで説明できる。

## Personas

### P1: 発表者兼開発者
- React/Viteアプリをcc-sddで作っている本人。
- 開発しながら、その過程を後で発表素材として見返したい。
- 主な関心は、開発の流れ、重要な判断、画面の変化、Viteらしい開発体験である。

### P2: 発表当日の発表者
- カンファレンス中に、安定したデモデータを使って説明したい本人。
- ライブ記録が壊れても、固定されたサンプルタイムラインで発表を継続したい。
- 主な関心は、迷わず見せられること、画像欠落やエラーがあっても話が止まらないことである。

### P3: ローカルプロジェクト管理者
- 仕様ファイル、実行時ログ、スクリーンショットを整理しておきたい本人。
- `.kiro/specs/` が大きくなりすぎたり、生成物が仕様Markdownに混ざったりすることを避けたい。
- 主な関心は、保存先の分離、ローカル完結、外部送信しないことである。

## MVP User Journey

1. 発表者がReact/Viteアプリの開発を始める。
2. cc-sddでrequirements、design、tasksを更新する。
3. Vite dev serverで画面を確認し、ファイル変更やHMRが発生する。
4. SpecLens Timelineがそれらをイベントとしてローカルに記録する。
5. 発表者はタイムラインを週単位で開き、前週/次週へ移動しながら開発の流れを見る。
6. 重要なイベントを選び、詳細やスクリーンショットを確認する。
7. スクリーンショットが保存できなかった場合でも、イベント本文は表示され、「写真を保存できませんでした」と分かる。
8. 発表に使うイベントを選び、固定デモデータとして分離する。
9. 発表当日はライブ記録ではなく、安定したデモタイムラインを表示する。

## User Stories

### US-01: 週ごとに開発の流れを見返す

**As a** 発表者兼開発者  
**I want** 開発イベントを1週間ごとのページで見たい  
**So that** 開発の流れを発表ストーリーとして整理しやすくしたい

#### Scenario
- Given 開発イベントが複数の日付にまたがって保存されている
- When 発表者がタイムライン画面を開く
- Then 現在週のイベントが表示される
- And 前週、次週へ移動できる

#### Success Criteria
- 現在週、前週、次週を切り替えられる。
- 週の開始日と終了日が分かる。
- その週のイベント数、スクリーンショット数、エラー数が分かる。

**Related Requirements:** 1.1, 1.2, 9.1, 9.3

### US-02: イベントの詳細を確認する

**As a** 発表者兼開発者  
**I want** タイムライン上のイベントを選択して詳細を見たい  
**So that** なぜその出来事が重要だったかを説明できる

#### Scenario
- Given タイムラインにイベントが表示されている
- When 発表者が1つのイベントを選択する
- Then 種別、時刻、関連ファイル、状態、説明が表示される

#### Success Criteria
- イベント一覧から詳細へ移動できる。
- 関連する仕様、タスク、ファイルパスを確認できる。
- スクリーンショットがある場合は同じ詳細画面で見られる。

**Related Requirements:** 1.4, 1.5, 4.3

### US-03: cc-sddの進行を記録する

**As a** 発表者兼開発者  
**I want** requirements、design、tasksの更新をタイムラインに残したい  
**So that** cc-sddで作った過程を発表で説明できる

#### Scenario
- Given 発表者がcc-sddの仕様ファイルを更新している
- When requirements、design、tasksなどのファイルが作成または更新される
- Then SpecLens Timelineはファイル種別、パス、更新時刻をイベントとして記録する

#### Success Criteria
- 仕様ファイルの更新がタイムラインに出る。
- 仕様、設計、タスクの違いが表示上で分かる。
- 関連付けできないイベントも消えずに残る。

**Related Requirements:** 2.1, 2.2, 2.3, 2.5

### US-04: 重要な意思決定を残す

**As a** 発表者兼開発者  
**I want** 開発中の重要な判断を手動で記録したい  
**So that** 発表時に「なぜその設計にしたのか」を説明できる

#### Scenario
- Given 発表者が設計方針を決めた
- When 発表者が意思決定を記録する
- Then 決定内容、理由、関連仕様またはタスクがタイムラインに追加される

#### Success Criteria
- 手動で意思決定イベントを追加できる。
- 発表で重要なイベントとして目印を付けられる。
- 関連する仕様やタスクがあれば紐付けられる。

**Related Requirements:** 2.4, 9.4

### US-05: Viteの開発体験を記録する

**As a** 発表者兼開発者  
**I want** ViteのHMR、フルリロード、エラーをタイムラインに残したい  
**So that** Viteを使った開発体験を発表で具体的に見せられる

#### Scenario
- Given Vite dev serverが起動している
- When ファイル変更によりHMR、フルリロード、またはエラーが発生する
- Then SpecLens Timelineは発生時刻、種別、関連ファイルをイベントとして記録する

#### Success Criteria
- `vite_hmr`、`vite_reload`、`vite_error` の違いが表示される。
- 詳細情報が不足していてもイベント自体は残る。
- エラーが起きてもタイムライン表示は継続する。

**Related Requirements:** 3.1, 3.2, 3.3, 3.4

### US-06: 重要な瞬間のスクリーンショットを残す

**As a** 発表者兼開発者  
**I want** 重要な画面状態をイベントと一緒に保存したい  
**So that** 開発過程を文字だけでなく視覚的にも説明できる

#### Scenario
- Given スクリーンショット対象のイベントが発生している
- When 自動キャプチャまたは手動キャプチャが実行される
- Then 画面画像が保存され、対応するイベントに関連付けられる

#### Success Criteria
- 手動で「この瞬間を保存」できる。
- 保存された画像をイベント詳細から見られる。
- 画像は仕様Markdownではなく`.spec-lens/`側に保存される。

**Related Requirements:** 4.1, 4.2, 4.3, 5.2, 5.3

### US-07: 写真が保存できなくても情報を見失わない

**As a** 発表者兼開発者  
**I want** スクリーンショット保存に失敗しても、取れているイベント情報だけは見たい  
**So that** 画像保存の失敗で開発記録全体を失わないようにしたい

#### Scenario
- Given イベント情報は取得できている
- And スクリーンショット保存だけが失敗した
- When 発表者がそのイベントを見る
- Then イベント本文は表示される
- And 画像欄には「写真を保存できませんでした」と表示される

#### Success Criteria
- スクリーンショット失敗時もイベントは残る。
- 画像欄には失敗状態が表示される。
- 可能であれば失敗理由も表示される。
- 他のイベント閲覧は継続できる。

**Related Requirements:** 4.4, 4.5, 9.1, 9.2

### US-08: 発表に使うイベントを探す

**As a** 発表者兼開発者  
**I want** 検索、フィルタ、重要マークでイベントを探したい  
**So that** 発表で使う場面を短時間で見つけたい

#### Scenario
- Given 多数のイベントが保存されている
- When 発表者が検索語、イベント種別、重要マークを指定する
- Then 条件に一致するイベントだけが表示される

#### Success Criteria
- タイトル、説明、種別、関連パスを検索できる。
- イベント種別で絞り込める。
- 重要イベントを目印として扱える。
- 保存済みデータ自体は削除されない。

**Related Requirements:** 1.3, 9.3, 9.4

### US-09: 簡易チェック結果を発表に含める

**As a** 発表者兼開発者  
**I want** 仕様ファイルや検証状態の簡易チェック結果を見たい  
**So that** cc-sddの進行が健全だったかを発表で説明できる

#### Scenario
- Given 発表者が簡易チェックを実行する
- When チェック対象ファイルやタスク状態が確認される
- Then 成功、警告、失敗、スキップのいずれかで結果が表示される

#### Success Criteria
- チェック結果がタイムラインイベントとして残る。
- 判定理由と関連ファイルを確認できる。
- 対象がない場合は失敗ではなくスキップとして扱う。
- AIによるコード品質採点とは表示しない。

**Related Requirements:** 6.1, 6.2, 6.3, 6.4, 6.5

### US-10: ライブ記録と発表用デモデータを分ける

**As a** 発表当日の発表者  
**I want** ライブ記録とは別に固定デモデータを開きたい  
**So that** 当日のデモを安定して実施したい

#### Scenario
- Given 発表用の固定デモデータが作成されている
- When 発表者がデモデータ表示に切り替える
- Then ライブ記録とは区別されたタイムラインが表示される

#### Success Criteria
- ライブ記録かデモデータかが画面上で分かる。
- デモデータに含まれるスクリーンショットを表示できる。
- 画像が欠落していても他イベントは表示できる。

**Related Requirements:** 7.1, 7.2, 7.3, 7.5

### US-11: 発表用データを作成する

**As a** 発表者兼開発者  
**I want** ライブ記録から発表に使うイベントだけを固定データとして切り出したい  
**So that** 本番で不要なイベントや不安定なライブ状態に依存しないようにしたい

#### Scenario
- Given ライブ記録に多数のイベントが保存されている
- When 発表者が使いたいイベントを選んでエクスポートする
- Then 選択したイベントと関連画像が`demo-data/spec-lens/`に保存される

#### Success Criteria
- 選択したイベントだけをデモデータ化できる。
- 関連する画像も一緒に分離される。
- ライブ記録は上書きされない。

**Related Requirements:** 7.4, 5.4

### US-12: 生成データで仕様ファイルを大きくしない

**As a** ローカルプロジェクト管理者  
**I want** 仕様Markdownと実行時データの保存先を分けたい  
**So that** `.kiro/specs/` を軽量に保ち、生成物の肥大化を避けたい

#### Scenario
- Given タイムラインイベントやスクリーンショットが増えている
- When SpecLens Timelineがデータを保存する
- Then 仕様Markdownには埋め込まず、実行時データは`.spec-lens/`へ保存される

#### Success Criteria
- 仕様、設計、タスクは`.kiro/specs/spec-lens-timeline/`に置く。
- イベントログ、画像、レポートは`.spec-lens/`に置く。
- 発表用固定データは`demo-data/spec-lens/`に置く。
- 保存先へ書けない場合はエラーを表示する。

**Related Requirements:** 5.1, 5.2, 5.3, 5.4, 5.5

### US-13: ローカルだけで使う

**As a** 発表者兼開発者  
**I want** ログインや外部サービスなしで使いたい  
**So that** 発表準備をローカル環境だけで進めたい

#### Scenario
- Given 発表者がローカル開発環境でアプリを起動している
- When タイムライン記録やスクリーンショット保存を使う
- Then データは外部サービスに送信されず、ローカルに保存される

#### Success Criteria
- ログインなしで使える。
- クラウド同期、共有URL、チーム権限管理は初期版に含めない。
- イベントや画像を外部サービスへ送信しない。

**Related Requirements:** 8.1, 8.2, 8.3, 8.5

### US-14: 発表中に壊れたデータで止まらない

**As a** 発表当日の発表者  
**I want** 一部のイベントや画像が壊れていてもタイムラインを見続けたい  
**So that** 発表中にデモが止まらないようにしたい

#### Scenario
- Given 一部のイベントデータや画像参照が壊れている
- When 発表者がタイムラインを開く
- Then 読めるイベントは表示され、壊れた箇所だけが分かる

#### Success Criteria
- 破損イベントは他イベントと分離して表示される。
- 欠落画像は欠落状態として表示される。
- テキスト情報は画像読み込みを待たずに表示される。
- 記録中、停止中、エラー中の現在状態が分かる。

**Related Requirements:** 7.3, 9.1, 9.2, 9.5

## Story Priority

### MVPで必須
- US-01: 週ごとに開発の流れを見返す
- US-02: イベントの詳細を確認する
- US-03: cc-sddの進行を記録する
- US-05: Viteの開発体験を記録する
- US-06: 重要な瞬間のスクリーンショットを残す
- US-07: 写真が保存できなくても情報を見失わない
- US-12: 生成データで仕様ファイルを大きくしない
- US-13: ローカルだけで使う
- US-14: 発表中に壊れたデータで止まらない

### 初期版で入れたい
- US-04: 重要な意思決定を残す
- US-08: 発表に使うイベントを探す
- US-09: 簡易チェック結果を発表に含める
- US-10: ライブ記録と発表用デモデータを分ける
- US-11: 発表用データを作成する

### 余力があれば磨く
- 表示密度、文言、発表用の見せ方を磨く。

## Not User Stories for MVP

以下は初期版では扱わない。

- チームメンバーとして共有URLからタイムラインを見たい。
- 管理者としてユーザー権限を管理したい。
- 過去の任意時点のアプリを完全に再実行したい。
- AIに設計品質やコード品質を採点してほしい。
- 複数リポジトリを横断して開発履歴を分析したい。
