# Brief: spec-lens-timeline

## Status
この仕様は2026-06-20時点で、本命アプリ候補としては保留する。
理由は、発表の主軸を「記録アプリを作ること」ではなく、「別のReact/Viteアプリをcc-sddで作り、その開発過程を記録して見せること」に戻すため。

この文書は、将来「開発過程を記録・可視化するアプリ」を作りたくなった場合の参考として残す。
次に進めるべき作業は、新しいReact/Viteアプリ本体の題材をdiscoveryすることである。

## Problem
Viteカンファレンスで、cc-sddを使ってReact/Viteアプリを作る過程と、完成したアプリの技術的価値を半々で見せたい。
しかし通常の開発ログは、仕様決定、タスク進行、コード変更、HMR、画面変化が分断されており、あとから発表用のストーリーとして再構成しにくい。

主な利用者は発表者本人。将来的には、開発チームの仕様駆動開発レビューや、OSS/個人開発者の開発ログ共有にも広げられる余地を残す。

## Current State
プロジェクトはグリーンフィールドで、React/Viteアプリ本体は未作成。
`.kiro/` と `.agents/` は存在するが、仕様・設計・タスク・実装はまだない。

Vite 8ではRolldown統合、HMR、Plugin API、Dev ServerのWebSocket通信などを利用できるため、Viteを単なるビルドツールではなく、開発過程を観測する基盤として扱える。
一方、過去バージョンのアプリを完全に操作可能な状態で再現するには、各時点のビルド成果物や実行環境を保存する必要があり、1か月以内のMVPとしては重い。

## Desired Outcome
cc-sddでアプリを作る過程を、あとから時系列で見返せるローカル開発タイムラインを作る。

MVPでは、次の情報を1つのタイムラインに統合する。

- `requirements.md`、`design.md`、`tasks.md` などの仕様ファイル更新
- タスクの開始・完了、検証結果
- Codexとの重要なやりとりや意思決定
- ViteのHMR、フルリロード、エラーなどの開発サーバーイベント
- 重要イベント時のスクリーンショット
- ユーザーが手動で保存した「この瞬間」のスクリーンショット
- AIではなくルールベースの簡易チェック結果

発表では、このアプリ自身をcc-sddで作った履歴をタイムライン表示し、「開発プロセス」と「完成アプリ」の両方を同じデモで説明できる状態にする。

## Approach
「SpecLens Timeline」として、React/ViteアプリとViteプラグインを組み合わせる。

Viteプラグイン側は、開発サーバー中のファイル変更、HMR、フルリロード、エラー、カスタムイベントをローカルのイベントログに記録する。
Reactアプリ側は、そのイベントログとスクリーンショットを読み込み、タイムライン、フィルタ、イベント詳細、スクリーンショットプレビューを表示する。

仕様・設計・タスクは `.kiro/specs/spec-lens-timeline/` に置き、軽量なProject Memoryとして保つ。
実行中に増え続けるイベントログ、スクリーンショット、スナップショット、レポートは、プロジェクト直下の `.spec-lens/` に保存する。
発表用に固定して残したいサンプルデータだけは、`demo-data/spec-lens/` に分けてGit管理できる形にする。

スクリーンショットは、重要イベント時の自動保存と、ユーザーが明示的に押す手動保存を併用する。
完全な過去実行環境の再現は行わず、画面状態はスクリーンショットとして保存する。

Viteの技術的新規性は、HMRイベント、モジュール更新、Dev ServerのWebSocket通信、Viteプラグインによる観測という形でアプリ価値に直結させる。

## Scope
- **In**:
  - React/Viteアプリの新規作成
  - Viteプラグインによるローカル開発イベントの記録
  - cc-sdd関連ファイル更新の記録
  - タスク進行、検証結果、重要な意思決定のタイムライン登録
  - 重要イベント時のスクリーンショット自動保存
  - 手動の「この瞬間を保存」操作
  - タイムラインUI、フィルタ、イベント詳細、スクリーンショットプレビュー
  - ルールベースの簡易チェック
  - `.spec-lens/` への実行時データ保存
  - `demo-data/spec-lens/` への発表用サンプルデータ保存
  - 発表用のデモデータ作成
- **Out**:
  - クラウド同期
  - 共有URL
  - ログイン、ユーザー管理
  - 過去バージョンの完全な実行再現
  - AIによるコード品質採点
  - 複数リポジトリの横断分析
  - チーム向け権限管理

## Boundary Candidates
- **Event Capture**: Viteプラグイン、ファイルウォッチ、HMRイベント、cc-sdd関連イベントを収集する責務
- **Event Store**: `.spec-lens/` 配下にローカルJSON/NDJSONなどでイベントとスクリーンショット参照を保存する責務
- **Screenshot Capture**: 重要イベント時と手動操作時に画面を保存する責務
- **Timeline UI**: イベント一覧、絞り込み、詳細、スクリーンショット表示を担う責務
- **Rule Checks**: 仕様ファイルの有無、タスク未完了、検証未実施などをルールベースで検出する責務
- **Demo Story**: 発表で見せるイベント種別、サンプルデータ、流れを整理する責務

## Out of Boundary
- ブラウザ上で過去時点のアプリを操作可能にすること
- クラウドへイベントやスクリーンショットを送信すること
- GitHub、Slack、外部AIサービスなどへの連携
- セキュリティ・保守性・設計品質をAIで採点すること
- 汎用的な監査ログ基盤にすること

## Upstream / Downstream
- **Upstream**:
  - cc-sddの仕様ファイルとタスク進行
  - Vite Dev Server
  - Vite Plugin API
  - Vite HMR API
  - Reactアプリの開発中画面
- **Downstream**:
  - Viteカンファレンス発表用デモ
  - チーム向けの仕様駆動開発レビュー機能
  - OSS/個人開発者向けの開発ログ共有機能
  - 将来的なAI要約・品質評価機能

- **Adjacent**: なし

## Constraints
- 1〜2週間でMVPを作り、最大1か月で発表用に磨く。
- 発表は「cc-sddで作る過程」と「完成したReact/Viteアプリ」を半々で見せる。
- Viteの技術的新規性を主役にする。
- 初期版はローカル開発環境で完結させる。
- `.kiro/specs/spec-lens-timeline/` には仕様・設計・タスクなどの軽量なProject Memoryだけを置く。
- 実行時に増え続けるログ、画像、スナップショット、レポートは `.spec-lens/` に保存し、原則Git管理しない。
- 発表で固定して使うサンプルデータだけは `demo-data/spec-lens/` に分離する。
- 画面履歴はスクリーンショットで扱い、完全な過去実行再現はしない。
- 品質評価はAIではなくルールベースの簡易チェックに留める。
- Markdown成果物は日本語で書く。
